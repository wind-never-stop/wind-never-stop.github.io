后端项目推进标准

发表于 2021-11-19 分类于 软实力 阅读次数：

项目阶段概要

项目的阶段包含：项目准备、项目启动（非必须）、项目研发、生产验收

整体流程如下：

一、项目准备

1. 意义

明确项目目标，确认大体需求范围，避免做“无用”的项目。

2.负责人事项

2.1 项目背景沟通，项目目标明确

与上级及组内相关同学初步沟通项目背景，确认是否有必要立项。

明确项目目标，必须包含：

  • 业务型需求：要完成的功能点、达到的业务指标（日活、成本、GTV等）
  • 技术型需求：要完成的改造点、人效、性能指标（响应速度、容量分析等）、可用性指标

2.2 需求调研

  • 技术型需求需要项目负责人或相关技术同学来负责收集，收集结果与上级及组内相关同学确认。
  • 业务型需求由产品同学负责收集。

2.3 职责范围确认

与相关产品、技术同学线下沟通需求背景，初步确认项目干系人。

2.4 建立项目钉钉群

建立项目钉钉群，将项目相关人员拉进群。

2.5 分配项目准备阶段任务

如有需求收集任务，将需求收集任务拆解分配到相关责任人。任务必须指定结束日期。

3. 阶段产出

  • 立项结论
  • 项目钉钉群

4.模板与工具

  • 钉钉项目群及项目助手钉钉项目管理实践

二、项目启动

1.意义

项目启动仪式感，表明项目正式开始。明细具体需求细节。

2.负责人事项

2.1 项目启动会

预定会议室，发会议邀请，组织召开项目启动会，同步背景，讨论目标是否合理、大致解决方案、大致分工协作人、资源方、关键节点。
会后发出会议纪要，会议纪要需要包含以下几个要素：会议时间、会议地点、参会人员、会议议题及相关结论、关键时间点。
关键时间点包括：需求评审（如有）、技术评审、联调、测试用例评审（如有）、提测、UAT、上线。
如项目较小，可通过钉钉群确认或线下确认，但结论必须要同步到项目群中。

2.2 需求文档编写

业务型需求文档由产品经理编写。
技术型需求文档非必须，可将相当内容补充到技术设计文档中。

2.3 需求评审

预定会议室，发会议邀请，组织召开需求评审，会上评审需求细节，讨论需求合理性，给出最终结论。需求文档根据会议讨论结果修改完善。
需求较小的情况下也可线下确认，但结论必须要同步到项目群中。

3. 阶段产出

  • 项目启动会会议纪要
  • 需求文档

4.模板与工具

三、项目研发

1.意义

标准的研发流程能够尽可能降低项目失败的风险，保证研发成果是符合项目预期的。

2.负责人事项

2.1 制定系统设计方案

参考设计方案模板，项目负责人可亲力亲为也可交待核心开发进行设计方案的撰写，需要对整个流程设计了然于心。

2.2 方案评审

负责人组织技术评审会，需要做以下工作：

  • 会议室预定
  • 通知需要参会的技术相关方
  • 掌控会议整体进度，防止问题发散
  • 梳理会议纪要，列出问题点
  • 如评审不通过，需要重复2.1~2.2

2.3 分工排期

  • 明确模块、任务项、预估工时、起止时间、开发负责人、备注
  • 利用钉钉项目助手录入任务

PS：如果是倒排期项目，按照任务优先级进行排期，对于排期之外的任务可考虑寻求外援或者与需求方商量完成最小版本，减少开发量

2.4 方案执行

  • 与对应的开发对齐实现细节，确认与开发没有理解偏差

  • 明确开发规范及流程

2.5 研发进度跟踪

  • 输出周报，邮件&项目群同步项目干系人
  • 里程碑事件，邮件&项目群同步项目干系人
  • 如有必要，可组织每日站会同步研发进度

2.6 风险点识别及解决

  • 人力资源风险：开发人员请假、离职
  • 技术实现复杂度风险：技术实现过于复杂，初始估时不够

解决方案：

  • 能够内部消化的，不影响整体排期
  • 否则，同步需求方和相关依赖方，重新进行「分工排期」流程。

3.模板与工具

3.1 设计方案模板

3.1.1 全新或者重构项目

XXX技术系分

  • 业务流程图（这个很关键，输出你对业务的理解，保证你下面的设计是符合需求的）
  • 依赖拓扑图（上下游的依赖，识别这个依赖可以很好防止你开始coding了才发现还有依赖方没有通知，导致delay）
  • 物理拓扑图（依赖什么中间件、使用的关键技术组件）
  • 程序、数据流程图（可以程序、数据可以合到一起，描述数据的流向和关键接口的设计）
  • 关键模块设计
  • 表模型、接口设计
  • 预估QPS，TPS
  • 技术风险点
  • 可灰度、可回滚、可降级（灰度策略、回滚策略、降级方案）【关键】

3.1.2改进型项目

  • 新老系统处理流程对比（标识改进点，关键处理逻辑）
  • 外部服务依赖（如果有新的外部依赖，整理好要依赖的服务接口描述）
  • 关键模块设计
  • 可灰度、可回滚、可降级（灰度策略、回滚策略、降级方案）【关键】

3.1.3推进迁移型项目

  • 背景
  • 期望完成时间（最终截止时间）
  • 影响服务方列表（包含负责人、排期、完成状态）
  • 迁移方案

四、生产验收

1.意义

明确项目已经完成生命周期，可以交付使用

2.负责人事项

2.1 周知干系人

  • 邮件&项目群周知项目已上线
  • 通知需求方进行验收

2.2 总结与复盘

  • 提前预定会议室并周知相关人员（开发测试产品）
  • 做总结性发言，陈列成果
  • 主导复盘会议
  • 记录提出的问题和待改进事项
  • 项目群同步复盘会议纪要
