活动抽奖榜单实现

发表于 2021-11-19 分类于 java ， redis 阅读次数：

背景

  • 提供给用户的高性能接口
  • 返回活动当天抽中礼物价值总和靠前的 用户排行榜
  • 返回活动当天自己每日抽中的礼物价值总和

流程

  • 查询redis key=activityId+day 存在返回
  • 不存在
    • 查询es 排名 + 50预留排名，接口异常、数据异常则抛错结束
    • 正常返回
      • 异步写入redis，key=activityId+day
      • 返回数据
  • 查询上榜用户冻结状态，接口异常兜底为非冻结
  • 过滤冻结用户后返回榜单前N
  • 查询活动自己每日抽中的礼物价值总和

不足

请求量过大，缓存失效间隙场景会出现耗时尖刺，部分用户体验较差

优化

提供定时主动刷新缓存机制

使用定时任务获取es数据，定时刷新redis值，保证用户查询时，缓存中的值均为最新榜单，消除耗时尖刺

20220306补充

  • 一、活动抽奖排行榜展示：

    • 1、背景：迭代比心app；内部的使用币种是比心币，当时背景是做一个抽奖活动，主要的流程是：在app中消耗比心币进行抽奖。这个抽奖得到的礼物总金额是与原始花费金额不等值的。我们现在要做的就是统计每天抽中礼物价值最高的前5个用户，形成一个排行榜单，进行展示。
    • 2、我的职责：我的职责就是负责实现一个高性能高并发的排行榜单查询接口，当前是通过redis缓存来实现的，当请求进来时，会先查询缓存，未命中缓存，再查询ES组装数据，返回给客户端，并异步写入redis
      这个里面使用了redis 的String数据结果，key是活动的id+当天日期，设置过期时间为5秒，value存的是Json String（uid，rank，totalGiftMoney）
    • 3、问题：
      • 1）当redis中没有数据时需要去查es,如果es查询出现异常和写redis异常，需要抛出异常，不能将空的数据更新到redis中，否则会出现一段时间内查询时，榜单都为空的情况
      • 2）如果请求量过大，缓存失效的时刻，会出现耗时尖刺。此时可以用定时任务去更新redis中的数据
      • 3）redis key 为啥要引入时间？引入时间作为key的一部分，是要兼容从第二天的零点开始重新累计榜单，59分存入缓存 04 00重新累计 5秒：可以容忍的不一致的最长时间

  • 二、如何实现高并发|数据接口

    • 1、读接口：通过缓存实现,
      • 1）缓存位置：pc web客户端缓存 client -> CDN -> 接入层缓存nginx-> 服务应用缓存 jvm 内存-guava->分布式缓存（redis）->原始真实数据 db、es
      • 2）缓存模式：
        • 业务侧缓存
          request->cache->真实数据源 db\es
        • 缓存当成独立数据源-组件化
          request->cacheManager
          查询缓存未命中是，才更新缓存
          cache->真实数据源
          写入真实数据时,立即写入缓存
          真实数据发生变化->binlog->变更立即写入cache
定时任务去更新redis中的数据
          异步写缓存
