消息消费框架

发表于 2021-11-19 分类于 java 阅读次数：

回顾

必要性

1.公司系统经过了平台化迁移，一个业务动作的完整逻辑分别在两个topic中完成，增加了代码维护成本，降低了可读性
2.很多消费逻辑是切异步在单独线程池中执行，执行失败没有完善的监控，告警机制，存在许多没必要的异步
3.梳理消费代码，主要就是转发业务kafka,发app消费，上报风险，执行统计等，而当前消费代码堆叠，可读性差

解决方法

使用模版方法对消费代码进行重构，流量灰度接入

1.通过apollo开关开启状态可使进入重构逻辑消费。如果有问题，可使Apollo开关置灰度，取而代之走老代码
2.平台转发业务kafka时，会对其进行着色，业务kafka消费时检查到未着色的kafka才消费，避免重复消费
着色的必要性&一次下线的问题
只针对发布的时候，发布的时候有多台机器，是分批逐步部署的。如果A机器上已经是新的框架代码了，此时它会发出业务kafka消息，并且没有执行着色操作。此时，B机器上还未部署，还是老代码，此时由于老代码业务逻辑是分两次在两个topic中执行的，而A发出的消息会被B监听到，会造成有些业务逻辑被执行两次（重复消费）。所以，从新代码中转发出来的消息（已完成全部逻辑）如果加上一个着色标记，然后在老代码中，如果有该标记，则不走老代码的后续操作。
3.一次下线的问题：有服务发到一半，转化出的业务kafka没有着色，如果被老代码的机器监听到，会出现两次消费
所以分两次下线

删除Apollo开关和着色代码

为什么不在API中做完所有逻辑？而要使用消息中间件？
1.如果在API中完成所有操作，会使接口显得臃肿，而且处理时间变长
2.耦合性高。如果放在接口中做，则A调B，A要关联B，知道B的处理逻辑。如果使用发送消息的方式，只管发送就行，不管后续是谁要消费，降低了耦合性。
