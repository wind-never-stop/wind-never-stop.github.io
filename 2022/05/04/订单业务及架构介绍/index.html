订单业务及架构介绍

发表于 2022-05-04 分类于 java 阅读次数：

未完待续。

1.业务背景

从毕业以来，一直在从事订单交易方向的后端研发，历经两家公司，交易体量从小到大，见证了订单架构的演进变化，踩过不少坑，特此反思总结下来，希望未来的路能越走越远。

1.1电商交易订单

1.1.1核心流程
  1. 用户从PC、小程序、APP、门店收银机等终端设备浏览商品；
  2. 加入购物车或者直接购买下单，进入订单购买页，选择优惠券、积分、兑换券等提交订单，进入订单结算页面；
  3. 通过微信、支付宝、银联等第三方服务商完成支付，订单推送到商家订单侧，进入待发货环节；
  4. 商家通过实体物流发货，填写上传物流单号，订单流转到已发货；
  5. 用户在平台中可查看实时物流信息，到货验货，完成订单。
1.1.2系统指标

承接、预售、拼团、超级团、营销三宝、闪购、众筹等玩法

闪购：高峰期TPS每秒在300+

查询QPS在1000+

1.2娱乐陪练订单

1.2.1核心流程
  1. 用户从PC、小程序、APP、门店收银机等终端设备浏览大神技能；
  2. 选择技能确认，进入到订单结算页面并唤起支付；
  3. 通过比心币完成订单支付（比心币不足则通过微信、支付宝、银联等第三方完成充值在购买），进入待接单环节
  4. 大神收到下单提醒，确认接单，进入已接单环节；
  5. 大神在线服务完成，上传服务凭证，用户点击完成订单。
1.2.2系统指标

下单QPS在10+

订单查询在100+

2.如何做-通过架构解决

电商架构

2.2交易平台化

采用领域设计，形成核心域、计价域、纠纷域、交付域，各自负责边界内的事情。

  • 1）针对交易动作，采用DDD领域设计，拆分出核心、计价、纠纷、交付域，实现领域动作并发出领域事件

  • 2）领域设计中，DB作为存储介质，为了考虑拓展性，往往会预留很多字段json化的拓展字段
    这导致不能满足各种订单复杂查询搜索需求，采用CQRS，命令查询职责分离原则，搭建订单读中心，设计ES大宽表，

    开发ES同步服务将DB数据实时同步到ES大宽表
开发ES搜索服务，查询ES大宽表来满足业务需求

  • 3）异构数据源同步，服务故障，网络闪断等原因，可能会出现数据不一致，因此开发订单工具维护后台，提供ES搜索，巡检，手动同步等页面，降低维护效率
    拥有ES后，可以实现订单数据明细查看，数据报表等

  • 4）交易动作经过领域拆分后，服务调用链路复杂化，问题定位排查不便，因此开发日志服务，记录订单各个动作节点日志，错误场景记录钉钉告警，并对接到工具后台，加快问题排查效率

  • 5）订单的一些通用数据结构，如数据实体，消息实体，领域事件名等上下游系统，如陪练小前端，活动系统，用户系统，统计服务是很关心，将上述通用结构放入订单组件包，交易平台内外使用统一实体，提高了对接沟通效率

  • 6）概括

    DDD领域设计写入：base\price\refund\delivery
CQRS订单读中心：query\sync\search\statistics
组件包：component
交易日志告警：log
工具后台：tool-web

2.3接入层

小前台、大中台原则，隔离依赖，做业务的适配层

2.3.1陪练小前台

平台基于拓展性考虑，只会承接核心能力，像比心订单场景下完成各个动作后，需要给比心app用户和大神发订单卡片消息，累计经验值等非常业务化暂时无法沉淀的东西，参考大中台小前台的原则，搭建陪练小前台，通过监听平台领域实践来实现，支持需求的快速迭代，平台保持相对稳定

之前陪练服务，经过多人迭代，未规范化，出现同一个业务动作，即消费平台消息，又消费转发后的业务消息；针对消息消费场景，采用模板方法抽象出框架，模板统一cat打点，日志处理，陪玩的每个业务动作只需要实现框架的 构建上下文，发app消息，转发业务消息，上报风控，后置业务即可，代码清晰可读性强

2.3.2客服系统

搭建工单系统，用于处理用户纠纷

客服日常工作分为，IM会话，工单任务，投诉处理，IM是购买的环信云服务，任务依靠jira流转，线下处理投诉打款，不方便在功能上做迭代和拓展，因此搭建客服工单系统，实现工单任务的自动分配流转，投诉处理线上打款走审批，对接交易平台，处理完成订单，部分退款，作废订单，拒绝申诉等动作

2.3.3打赏系统

比心定位是游戏娱乐陪玩版块，打赏也是其中一种，在用户聊天页，用户动态页，直播和语言房等场景下，用户可以拉起礼物面板选择礼物，付费打赏给对方，对方获得收入，同时观看礼物动效。为吸引用户，开发如，幸运币、盲盒、连击、批量等打赏玩法，前期支持业务快速上线，在开发测放了大量的配置，活动经常做，开发测试成本比较高，开发沉淀出打赏活动后台，提供活动、礼物概率等一系列配置，交付给运营使用，目前运行1年多，承接10+活动，幸运币抽奖礼物打赏玩法最高单日达到1200万流水

打赏系统owner，负责需求分析，方案设计，项目跟进

3.困难阻碍

遇到的最大的困难阻碍是什么？

  1. 领导短期看不到价值，不认可这次重构
  2. 需要求产品、测试同学，不了解这个领域模型
  3. 组内同事缺乏，领域设计能力，沟通存在较大隔阂
  4. 为提升性能，用了哪些优化手段？
  5. 为提升可用性，用了哪些优化手段？

4.个人收获

了解了订单交易方向的架构演进，可以在以后的工作中，结合业务实际情况，选用不同的架构，在性能达标的情况下，在三到五年内保留一定的拓展性，完成业务既定目标。
