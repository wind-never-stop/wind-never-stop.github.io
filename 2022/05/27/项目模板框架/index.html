项目模板框架

发表于 2022-05-27 分类于 互联网 阅读次数：

1.背景

团队在搭建的过程中，经常会遇到大家立即不一致，对一些基础的东西产生分歧，沟通对接成本比较大，如何提高沟通效率，为需求迭代增速，并维持敏捷开发，这是一个软件开发部门需要面对的命题

2.结论

形成团队最佳时间规范，信息对齐，基本原则对齐，输出项目代码模板，接口规范等等最佳实践的case，以达到软件的敏捷快速迭代并保持稳定性，简单原则。

  • 统一的项目代码模板，项目分层
  • 接口规范（含异常处理规范）
  • 数据库规范 mysql
  • 调度平台规范（含定时调度、延时调度）
  • 日志查询规范 elk
  • 消息生产消费框架

3.1.2微服务架构-dubbo

注释说明：

  • api，提供给dubbo调用解析，每个接口必须有对象，以 xxxRequest/xxxResponse结尾，存在接口层复用属性对象以xxxDTO结尾，返回结果用基础包 ado-common-base Response包装
  • serviceImpl，很薄的一层，只做dubbo provider暴露，参数组装，请求路由，一个method对应一个Manager
  • manager，业务流程组装，流程编排，流程中有公用逻辑抽取出，commonManager来处理，可以调用commonManager处理，尽量避免业务manager之间互相调用；
  • handler，最小执行单元，比如，对外api调用，公用算子等，handler之间不允许互相调用；
  • repository，是特殊的handler，做数据持久化，事务控制，do作用域终止在这一层，bo(business object)和do(data object)互转，json化字段拆装箱等；
  • Mq Producer，是特殊的handler，做消息生产；
  • Mq Consumer，很薄的一层，只做消息消费路由，业务流程在manager中完成。
3.1.3单体项目模板-web

注释说明：

  • web controller，很薄的一层，每个接口必须有对象，以 xxxRequest/xxxVO结尾，存在接口层复用属性对象以xxxVO结尾，返回结果用基础包 ado-common-base Response包装
  • adapter，请求响应组装，基本参数校验，设计db查询、接口调用请下沉到service处理
  • service，业务逻辑层，考虑到B端存在简单业务逻辑，也可以在service中直接实现，不走manager，一个method对应一个Manager
  • manager，业务流程组装，流程编排，流程中有公用逻辑抽取出，commonManager来处理，可以调用commonManager处理，尽量避免业务manager之间互相调用；
  • handler，最小执行单元，比如，对外api调用，公用算子等，handler之间不允许互相调用；
  • repository，是特殊的handler，做数据持久化，事务控制，do作用域终止在这一层，bo(business object)和do(data object)互转，json化字段拆装箱等；
  • Mq Producer，是特殊的handler，做消息生产；
  • Mq Consumer，很薄的一层，只做消息消费路由，业务流程在manager中完成。
3.1.4项目脚手架

通过模版新建项目，infoq原理解释maven模板插件使用

生成步骤：

  1. 下载解压到本地目录。（注 ：目录名不能包含中文）
  2. idea中安装插件Maven Archetype Catalogs （搜索时 用Maven作为关键字。其他关键字无法找到该插件）重启 idea
  3. 打开项目 ，使用私服的 maven settings 文件。
  4. 在模板的项目中运行 mvn archetype:create-from-project，会在target目录下生成一个archetype目录，这其实是一个脚手架的maven项目
  5. 然后切到archetype目录下: cd target/generated-sources/archetype/,然后对将脚手架安装到本地maven仓库中: mvn install
  6. 这时去到maven 仓库中可以看到archetype-catalog.xml 文件中已经添加新的模版。（注意这里是配置的maven 仓库位置。如果本地有多个maven版本和仓库需要自己找到 idea 配置文件中的那个）
  7. 添加模板到IDEA。（第二步安装的 maven archetype catalogs 插件作用）archetype-catalog.xml 添加到插件中。重启idea。
  8. 生成自己的项目，执行以下命令,即可在当前目录生成符合本标准的项目结构代码

3.4调度平台规范

项目中会存在各种各样的业务定时任务,但是大部分任务都需要自己写一套业务打点和日志打印的代码，重复代码较多(有的甚至没有写打点)

针对这种情况，单独定义了几个抽象类，以后大家在写定时任务时按照功能不同直接继承即可。

抽象类说明

抽象类名称 场景说明 举例
AbstractPlainJob 简单的job,建议场景:1.无须扫描数据库,直接调用业务方的job2.没有任何条件，直接执行的job…
AbstractPlainAsyncJob 简单异步执行的job
AbstractDisposableJob 一次性的job,建议场景:1.修复数据的job;2.只会运行一次的job;… TaskCompensateJob
AbstractComplexJob 复杂的job,建议场景:1.需要先扫描数据库,然后调用业务方接口(参数是单条数据)的job…
AbstractComplexBatchJob 复杂的批量执行的job,建议场景:1.需要先扫描数据库,然后调用业务方接口(参数是批量数据)的job…
AbstractComplexStepJob 复杂的需要使用步长的单条执行的job,建议场景:扫描一个范围的数据，但范围太大，需要分割成更小的单元后单条执行 … TaskJob
AbstractComplexStepBatchJob 复杂的需要使用步长的批量执行的job,建议场景:扫描一个范围的数据，但范围太大，需要分割成更小的单元后批量执行 …
BatchProcessManager 工具Bean，大量数据需要按时间窗口移动分批拉取数据，做单条处理的场景
