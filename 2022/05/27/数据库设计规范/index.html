数据库设计规范

发表于 2022-05-27 分类于 最佳实践 阅读次数：

通用设计规范

  • 【强制】数据库中重要的对象要有注释：

    • 每个表都要有注释；
    • 字段除了常见的自增主键id, create_time, update_time，其他字段必须要有注释；
    • 索引等也建议有注释；
    • 所有注释要保持最新；

  • 【建议】默认使用utf8mb4字符集，无乱码风险。utf8mb4 是 utf8 的超集，可以获得比 utf8 更好的兼容性。

  • 【建议】除非 DBA 建议，存储引擎一律使用 INNODB；

  • 【强制】数据库时区设置为UTC时区；

  • 【强制】禁止使用数据库的外键特性，在应用层寻求对应的解决方案；

  • 【强制】禁止使用数据库的触发器、存储过程 及 数据库的函数，有需要，在应用层做相关处理；

  • 【建议】尽可能降低对 Mysql 独有特性的依赖，因为这种特性未来可能带来更大的迁移成本（例如：假如后续从 mysql 迁移至 postgresql）；

  • 【建议】数据库表中设置三个字段：id（主键）、create_time（记录创建时间）、update_time（记录更新时间）；

  • 【强制】更新数据表记录时，必须同时更新记录对应的 update_time 字段值为当前时间；

  • 【强制】id类型没有特殊要求，必须使用bigint unsigned，禁止使用int，即使现在的数据量很小。id如果是数字类型的话，必须是8个字节；

  • 【建议】除数据库表主键id之外，建议设置业务数据主键（例如content_id, order_id等），该类型为字符串，对外暴露业务主键，防止从接口遍历数据；

  • 【建议】所有的数据库表的字段都应该是 not null 的，并且配置默认值。

  • 【建议】字段长度定义应考虑业务场景，满足需求的前提下应尽量短，以提高查询的效率，且降低索引对资源的消耗。

  • 【强制】不同表中存储相同数据的列名和列类型必须一致（注：一般作为关联列，如果查询时关联列类型不一致会自动进行数据类型隐式转换，会造成列上的索 引失效，导致查询效率降低）

  • 【建议】经常一起使用的列应该放到一个表中，允许适当冗余，避免更多的关联操作。字段允许适当冗余，以提高查询性能，但必须考虑数据一致。冗余字段应遵循：

    • 不是频繁修改的字段。
    • 不是唯一索引的字段。
    • 不是 varchar 超长字段，更不能是 text 字段。

  • 【建议】单库表数量建议控制在 50 以内；

  • 【建议】表列数量建议控制在30个以内；

  • 【建议】单表行数超过 500 万行或者单表容量超过 2GB，才推荐进行分库分表。 说明:如果预计2年后的数据量根本达不到这个级别，请不要在创建表时就分库分表；

  • 【建议】尽量做到冷热数据分离，减小表的宽度；

  • 【建议】数据行的长度要小于物理页一半的大小（默认情况下，页大小 16 KB，则半页大小是 8 KB，最大行长度不能超过 8000 Byte，具体可以参考 mysql 官方文档），如果超过这个长度的话在物理页中插入两行数据就会出现行链接从而造成存储碎片，降低查询效率；

  • 【建议】数据库链接池的上下限设定遵循此规则：

    • 下限：服务运行实例的CPU核数+1

  • 上限：不高于（真实业务场景QPS上限 * 单请求响应时间(s) / 服务运行实例数）。例如：单请求响应时间全局参考值可取0.05，对业务场景1000QPS，服务运行实例为5的应用，配置连接池上限为：1000*0.05/5=10

  • 【强制】禁止在数据库上存储图片、二进制文件等静态资源，应该使用合适的文件系统，数据库仅存储URL对于二进制多媒体数据、超大文本数据也不要放在数据库字段中；

  • 【强制】禁⽌在数据库中存储明文密码；

命名规范

  • 【强制】禁止使用拼音来命名，禁止拼音英文混用

  • 【强制】表名、字段名必须使用小写字母或数字，使用下划线分隔单词。不要使用大写字母，禁止出现数字开头，禁止两个下划线中间只出现数字（注：Mysql在Windows下不区分大小写，在Linux下表名字段名等区分大小写，为了避免不必要的麻烦，命名一律使用小写）

  • 【强制】表名不使用复数名词，而且要简明扼要，尽量控制在64个字符以内，但不应以缩短表名为由，使用辨识度不高的缩写。

  • 【强制】数据库的命名建议：{应用名称}{region}{env}。例如：account_eu_prod

  • 【建议】表的命名最好是加上“业务名称_表的作用”

  • 【强制】禁用保留字，如 desc、range、match、delayed 等，请参考 MySQL 官方保留字

  • 【强制】表达是与否概念的字段，使用 is_xxx 的方式命名，数据类型是 unsigned tinyint(1)（1 表示是，0 表示否），举例：is_deleted

  • 【强制】主键索引名添加前缀”pk_”，唯一索引名前缀 “uniq_”，普通索引名前缀 “idx_”

类型规范

  • 【强制】对于tinyint类型，如果表示boolean类型，需要设置为tinyint(1)；如果表示0-255（-128-127）之间的整数，需要设置为tinyint(4)

  • 【建议】尽量少使用TEXT、避免BLOB数据类型，最常见的TEXT类型可以存储64k的数据

  • 【建议】避免使用ENUM类型，可使用 tinyint (数量比较多的就用 small int 或者 直接用 int)代替。因为修改 ENUM 需要使用 ALTER 语句，需要进行 DDL 操作， ENUM 类型的 ORDER BY 操作效率低，需要额外操作。tinyint 枚举的可能取值在comment里面标明。

  • 【强制】禁止使用 float、double 来定义财务类（例如金额）字段，应该使用 decimal 类型或者bigint类型。金额字段使用 decimal 类型，并给予足够的长度及精度。在性能要求比较苛刻的情况下，使用 bigint 类型，单位是分（如果是其他货币，需要定义其他单位）

  • 【建议】如果存储的字符串长度几乎相等（如用户名、密码等长度变化不大的字段），使用 char 定长字符串类型，其他不确定长度的字段，统一使用varchar相关的类型

  • 【建议】尽量少使用 varchar 类型的大字段，减少行迁移的发生。varchar 长度不要超过 5000，如果存储长度大于此值，定义字段类型为 text，独立出来一张表，用主键来对应，避免影响其它字段索引效率

  • 【建议】尽量使用整型字段（明确无符号的数值，使用 unsigned的整型），代替 IP、枚举类型、字符类型、浮点数类型（ip 地址的每段可以看成是一个 0-255 的整数，把每段拆分成一个二进制形式组合起来，然后把这个二进制数转变成一个长整数。 组合起来即为：00001010 00000000 00000011 11000001,转换为10进制数就是：167773121，即该 IP 地址转换后的数字就是它了。）

  • 【强制】时间字段使用时间日期/时间戳类型，不要使用字符串或者bigint类型存储

索引规范

  • 【建议】限制每张表上的索引数量，建议单张表索引不超过5个

  • 【禁止】禁止给表中的每一列都建立单独的索引

  • 【强制】业务上具有唯一特性的字段，即使是多个字段的组合，也必须建成唯一索引

  • 【强制】超过三个表禁止 join。需要 join 的字段，数据类型必须绝对一致; 多表关联查询时， 保证被关联的字段需要有索引

  • 【强制】页面搜索严禁左模糊或者全模糊，如果需要请走搜索引擎来解决

  • 【建议】建组合索引的时候，字段顺序应该考虑一下三个因素：字段的区分度，字段的使用频繁程度，字段长度大小

  • 【建议】禁止在区分度不高的属性上建立索引（此处是广义的区分度，比如进行中状态的订单占比很少，而应用基本只查询进行中的订单，则可以在订单状态上加索引）

  • 【建议】建立长字符串字段的前缀索引，保证快速有效过滤数据的同时，节省维护索引的开销。

SQL规范

  • 【强制】禁止在代码中使用跨库查询；
  • 【强制】超100万行的批量写（UPDATE、DELETE、INSERT）操作，要分批多次进行操作，缓解主从延迟
  • 【强制】对于大表使用pt-online-schema-change修改表结构
  • 【强制】对于程序连接数据库账号，遵循权限最小原则
  • 【建议】避免物理删除，尽量采用逻辑删除
  • 【建议】避免删除字段
  • 【强制】禁止使用存储过程，存储过程难以调试和扩展，更没有移植性
  • 【强制】禁用select *，必须指定列
  • 【建议】减少与数据库交互次数，尽量采用批量SQL语句
  • 【强制】禁止在主库上执⾏后台管理和统计类功能的QUERY，必要时申请统计类从库

操作行为规范

  • 【强制】禁止在线上做数据库压力测试；
  • 【强制】禁止从开发环境，测试环境直接连接生产环境数据库；
  • 【强制】禁止在数据库事务开启后处理非必要的逻辑，禁止在事务开启后访问非本地资源；
  • 【建议】如果修改字段含义或对字段表示的状态追加时，需要及时更新字段注释；

数据库账号规范

  • 【强制】禁止为程序使用的账号赋予super权限
  • 【强制】业务程序必须实现读写分离，读写账号以后缀_w命名，只读账号以后缀_r命名
  • 【强制】程序账号权限只能是SELECT、INSERT、UPDATE、DELETE，除非特殊情况可以授予CREATE和DROP权限
  • 【强制】不同后端平台以及运维程序，需要使用不同的数据库账号
  • 【建议】相同数据库账号的权限必须保持一致；主从数据库的账号授权保持一致

常用字段名建议

数据库表通用字段命名

字段名 含义 推荐类型 备注
id 数据库主键 BIGINT(20) UNSIGNED
create_time 创建时间 timestamp 默认 CURRENT_TIMESTAMP
update_time 更新时间 timestamp 默认 CURRENT_TIMESTAMP
country 国家名称 varchar(90) 例如：Norway
city 城市名称 varchar(189) 例如：Oslo
zip_code Zip Code varchar(18)
country_code 国家编码 char(2) ISO 3166-1 alpha-2 (例如 CN, NO)
region 地区/区域 varchar
lang 语言 varchar() LANGUAGE是mysql保留关键字，故使用 lang。此字段取值例如：en_US, zh_CN, …
email 邮箱地址 varchar(254) See RFC5321
address 地址 varchar(255) 海外大部分的地址是分 address1（详细地址）, address2（补充地址）
account_id Nio Account ID int unsigned 绝大部分系统在存储用户相关数据的时候都需要使用account_id。受限于车机系统，目前account_id是32位无符号整形数
user_id User id varchar(20) 用户，创建nio account之前的用户标识。只在少数几个场景(e.g. leads)下使用，其他场景都使用 account_id
first_name 名 varchar(50)
last_name 姓 varchar(50)
latitude 位置 - 纬度 decimal(10,6) -90.000000 ~ 90.000000
longitude 位置 - 经度 decimal(10,6) -180.000000 ~ 180.000000
status 状态 tinyint 根据业务场景设置位数

其他推荐命名

字段名称 适用场景 实体描述 易混淆词
vehicle 通用场景 用于被服务的车辆，一般用于客户车 car（一般不使用这个词）
reservation 预约了某一特定资源一段固定的时间段的场景 一般用于描述某一特定资源在某一个特定时间段的占用情况 booking
original_XXX_time 时间强相关的场景 XXX 的原始（既定）时间 –
estimate_XXX_time 时间强相关的场景 XXX 的（最新）预估时间 –
actual_XXX_time 时间强相关的场景 XXX 的实际时间 –
staff 涉及专员管理系统（oathkeeper）提供的服务人员的场景 专员（俗称小哥） –
dispatch 派单场景 分配任务 assign（dispatch 的客体是任务，assign 的客体是执行任务的人）

supplier 对接了不同的下游服务供应商 一般用于区分对接的不同的下游供应商 channel
