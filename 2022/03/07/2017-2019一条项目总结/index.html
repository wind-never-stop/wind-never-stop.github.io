分布式锁做防重复提交订单：
    1）渲染订单下发submitKey，提交订单传递submitKey --> 表达页面过期
    2）redis: set("nx", "ex", expireTime)->"OK" --> 获取分布式锁成功
    3）zookeeper: 临时目录节点 --> 创建节点
        判断刚创建的节点是否为第一顺序节点 
        --> true：加锁成功
        --> false: 注册一个回调监听前一个节点的释放动作
            （回调内容，判断自己创建的节点是否为第一顺序节点-> true:加锁成功）

分布式事务实现最终一致性：tcc:try->confirm->cancel 
    1) try方法上提供注解：指定事务完成（confirmMethod）或回滚需要调用的方法(cancelMethod)
    2) AOP实现：
        a) 调用try方法后，调用tcc.uploadStatus dubbo接口（状态保存到redis list：lpush lrange lset del scan，根据try是否抛异常来异步流转事务状态）
        b) 根据主调方 try方法是否抛异常来决定 tcc call confirmMethod or cancelMethod，注意这两方法会要求传入 transId, branchId
    3）举例：以库存接入tcc事务为例
        a) 声明并实现 try，注意服务按需自行保存 事务提交 or 回归的必要参数，也就是 transactionLog（一般被调方需要保存）
        b) 声明并实现 confirm 和 cancel 方法，根据传入的 transId, branchId 从 transactionLog 获取必要参数进行后续逻辑
        c) spring配置文件声明 AspectBean 启用 tcc aop

高并发扣减库存：库存存放于redis，如何保证最终一致性:记param-log
    1）try阶段：pipeline.multi()-->submit()
        save reduceStockParamsLog to redis 
        reduce redis skuStock
    2）confirm阶段：delete reduceStockParamsLog
    3）cancel阶段：pipeline.multi()-->submit()
        return redis skuStock
        delete reduceStockParamsLog

Redis 的事务保证了 ACID 中的一致性（C）和隔离性（I），但并不保证原子性（A）和持久性（D）
    如何保证一致性：入队错误、执行错误、Redis 进程被终结 三个话题来讨论
    为何不支持持久性：
        a）在单纯的内存模式下        
        b) 在 RDB 模式下，服务器可能在事务执行之后、RDB 文件更新之前的这段时间失败
        c) 在 AOF 的“总是 SYNC ”模式下，事务的每条命令在执行成功之后，都会立即调用 fsync 或 fdatasync 将事务数据写入到 AOF 文件。但是，这种保存是由后台线程进行的，主线程不会阻塞直到保存成功，所以从命令执行成功到数据保存到硬盘之间，还是有一段非常小的间隔，所以这种模式下的事务也是不持久的。

订单系统-下单框架-TradeGroup-反转控制
1）贫血模型：合法性校验、各个玩法计算逻辑全部由订单自己控制计算，需要的基础数据从玩法系统提供基本的query接口获取，随着玩法的层出不穷，订单分支越发复杂，一个小改动，就会牵一发而动全身，贴时序图

2）丰富模型：下单框架提供下单步骤，由玩法系统组装下单流程，构建TradeGroup，订单基于TradeGroup，调用下单各阶段组件完成下单
    举例：拼团下单：商品合法性校验、扣库存、用券、用积分、取消订单时间、发用户通知消息，等等
    a) InnerAction().doAction():
        TradeActivityGroup：由玩法系统组装配置

dubbo接口调用

    2）根据请求传入的method, 从ApiSchema对象中获取HttpApiExecutor，执行dubbo调用
    3）依赖调用、组合调用：工作流 WorkflowGraph-Node(链表)-Stage（森林）-Execution

API接口文档生成：
    1）启动时解析api.jar、加载api.class进jvm、并基于api上的注解内容，生成ApiMethodInfo,并将其维护在ApiSchema 对象中
    2）提供查询接口，返回所有服务端可用的ApiMethodInfo.json（Api接口文档）
